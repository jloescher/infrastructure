### 1. Fixed Laravel SQLite Default Issue

**Problem**: Laravel apps were defaulting to SQLite instead of PostgreSQL.

**Root Cause**: The `run_framework_setup` function used `sed` to replace DB_* values, but:
- `sed` only replaces existing lines
- If `.env.example` had `DB_CONNECTION=sqlite`, it stayed
- If DB_* variables didn't exist, they weren't added

**Fix** (`app.py:513-560`):
- Explicitly set `DB_CONNECTION=pgsql`
- Use `grep -q` to check if variable exists, append if missing
- Re-run `php artisan key:generate --force` after env updates

## Phase 6: Database Permissions + Deploy Script Fix (Added 2026-03-17 ~21:15 EDT)

### 1. PostgreSQL Schema Permissions

**Problem:** Laravel migrations failed with `permission denied for schema public`.

**Root Cause:**
- PostgreSQL 15+ changed default permissions on the `public` schema
- App users were created but lacked CREATE TABLE permissions
- Issue surfaced after database recreation

**Fix:** Granted schema permissions on PostgreSQL primary:
```sql
GRANT ALL ON SCHEMA public TO rentalfixer_user;
GRANT ALL ON SCHEMA public TO rentalfixer_admin;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO rentalfixer_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO rentalfixer_admin;
```

**Result:** Deploy progressed past migrations, but revealed next issue.

### 2. Deploy Script Backup Bug

**Problem:** `pg_dump: error: too many command-line arguments (first is "rentalfixer_staging_user")`

**Root Cause:** The `backup_database()` function in `/opt/scripts/deploy-app.sh` uses:
```bash
DB_USER=$(grep DB_USERNAME .env 2>/dev/null | cut -d'=' -f2)
```

This matches **both** `DB_USERNAME` and `STAGING_DB_USERNAME`, causing pg_dump to receive multiple values.

**Fix Required:** Change to anchored grep:
```bash
DB_USER=$(grep "^DB_USERNAME=" .env 2>/dev/null | cut -d'=' -f2)
```

All four grep patterns in `backup_database()` need fixing:
- `DB_NAME=$(grep DB_DATABASE ...)` → `grep "^DB_DATABASE="`
- `DB_USER=$(grep DB_USERNAME ...)` → `grep "^DB_USERNAME="`
- `DB_PASS=$(grep DB_PASSWORD ...)` → `grep "^DB_PASSWORD="`
- `DB_HOST=$(grep DB_HOST ...)` → `grep "^DB_HOST="`

### 3. Cache Table Missing (Expected)

**Problem:** `cache:clear` command failed because `cache` table doesn't exist.

**Root Cause:** Migrations haven't run yet (blocked by backup bug), so cache table doesn't exist.

**Status:** This is expected behavior - the script has `|| true` to continue. Will resolve once backup bug is fixed.

## Phase 7: Database Schema Permissions in Creation Flow + Secrets Wizard Enhancement (Added 2026-03-17 ~21:30 EDT)

### 1. Schema Permissions Must Be in Database Creation Flow

**Problem:** Schema permissions granted manually are lost when databases are recreated.

**Root Cause:**
- PostgreSQL 15+ restricts `public` schema by default
- The database creation code in `app.py` grants database-level privileges but not schema-level
- When a database is deleted and recreated, manual grants are lost

**Required Fix:**
After creating each database, connect to it and grant schema privileges:

```python
def grant_schema_permissions(db_name, db_user, db_admin):
    conn = psycopg2.connect(
        host=PG_HOST, port=PG_PORT, user=PG_USER,
        password=PG_PASSWORD, database=db_name
    )
    conn.autocommit = True
    cur = conn.cursor()
    
    cur.execute("GRANT ALL ON SCHEMA public TO {};".format(db_user))
    cur.execute("GRANT ALL ON SCHEMA public TO {};".format(db_admin))
    cur.execute("ALTER DEFAULT PRIVILEGES FOR ROLE {} IN SCHEMA public GRANT ALL ON TABLES TO {};".format(db_admin, db_user))
    cur.execute("ALTER DEFAULT PRIVILEGES FOR ROLE {} IN SCHEMA public GRANT ALL ON SEQUENCES TO {};".format(db_admin, db_user))
    
    cur.close()
    conn.close()
```

## Phase 8: Database Creation Flow Bug Fix (Added 2026-03-17 ~22:00 EDT)

### Problem 1: "cursor already closed"

**Symptoms:**
- Database creation fails with `cursor already closed` error
- Staging database/users not created

**Root Cause:**
```
Line 1872-1873: cur.close() + conn.close()  ← Closes after production DB
Line 1875:     grant_schema_permissions()   ← Opens own connection (OK)
Line 1892:     cur.execute("CREATE USER...") ← ERROR! Uses closed cursor
```

The connection was closed AFTER production database creation but BEFORE staging user creation. When staging block tried to use `cur`, it was already closed.

### Problem 2: "Missing required deploy secrets"

**Symptoms:**
- Deploy fails with `Missing required deploy secrets for production: DB_USERNAME, DB_PASSWORD`
- Secrets not saved even though users were created

**Root Cause:**
1. DB credentials stored in `results` dict at end of try block
2. If exception occurs (like cursor error), `results` is empty
3. `set_app_secret()` calls check `results.get("db_user")` which returns None
4. Secrets never saved

### Solution: Reorder Flow

**New Flow:**
1. Create production users → **Save production secrets immediately**
2. Create production database → Grant schema permissions
3. Create staging users (if enabled) → **Save staging secrets immediately**
4. Create staging database (if enabled) → Grant schema permissions
5. Close connection (only once, at the end)
6. Continue with application creation and deployment

| File | Changes |
|------|---------|
| `dashboard/app.py` | Remove premature close, save secrets early, single close at end |

## Phase 10: Laravel Deploy Fixes + Permission Hardening (2026-03-17 23:30-23:55 EDT)

### Problem

Rentalfixer app deployment failed with "Primary failed" error. Investigation revealed multiple issues:

1. **PHP-FPM Pool Config Syntax Error**: `php_admin[` directive is invalid, should be `php_admin_value[`
2. **Missing Laravel Setup**: `run_pull_deploy()` didn't create nginx/PHP-FPM configs for new Laravel apps
3. **.env Permissions**: Created as `webapps:webapps 640`, but PHP-FPM runs as `www-data` which couldn't read it
4. **Health Check Port Bug**: Deploy script extracted port from PHP-FPM config (which uses sockets) instead of nginx
5. **JavaScript Error**: `create_staging` element null check missing

### Root Cause Analysis

**PHP-FPM Config:**
```
# WRONG (line 152 in app.py)
php_admin[value[disable_functions] = exec,passthru,shell_exec,system

# CORRECT
php_admin_value[disable_functions] = exec,passthru,shell_exec,system
```

**Laravel Setup Flow:**
- Initial deployment skipped `setup_laravel_app()` because it was only called during app creation
- `run_pull_deploy()` assumed nginx/PHP-FPM configs already existed
- For first-time deploys via dashboard, these configs weren't created

**Permission Model:**
```
# .env created as:
webapps:webapps 640  → www-data cannot read

# Should be:
webapps:www-data 640 → www-data can read
```

### Permission Model (Documented)

| Path | Owner | Group | Mode | Purpose |
|------|-------|-------|------|---------|
| `/opt/apps/{app}` | webapps | webapps | 755/644 | Code world-readable for www-data |
| `storage/` | webapps | www-data | 2775 | Setgid ensures new files inherit group |
| `bootstrap/cache/` | webapps | www-data | 2775 | Same |
| `.env` | webapps | www-data | 640 | www-data needs read, not world-readable |

### Lessons Learned

1. **Always test new deployments end-to-end** - The PHP-FPM syntax error only surfaced on actual app deployment
2. **Permission race conditions are fragile** - Set correct permissions immediately on file creation, not just in cleanup scripts
3. **Health checks must match architecture** - PHP-FPM uses sockets, nginx uses ports; check the right config
4. **Null checks for optional DOM elements** - Not all form paths render all elements
