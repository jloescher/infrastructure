## Critical Fix: HAProxy Consolidated Frontend Architecture

### Problem
Both production (rentalfixer.app) and staging (staging.rentalfixer.app) were returning 503 errors through Cloudflare, while working fine on direct router access.

### Root Cause Analysis
1. HAProxy had separate frontends for each domain on port 443
2. Multiple frontends binding to the same port with SSL caused SNI routing issues
3. HAProxy's SNI-based frontend selection was inconsistent
4. Requests were not matching the correct frontend's ACLs

### Investigation Steps
```bash
# Direct router tests showed different results
curl -sk --resolve rentalfixer.app:443:172.93.54.112 https://rentalfixer.app  # 200
curl -sk --resolve rentalfixer.app:443:23.29.118.6 https://rentalfixer.app    # 503

# HAProxy stats showed backends were UP
echo 'show stat' | socat stdio /run/haproxy/admin.sock | grep rentalfixer

# But localhost tests showed the issue
curl -sk --resolve rentalfixer.app:443:127.0.0.1 https://rentalfixer.app  # 503
```

### Solution
Changed from **per-domain frontends** to **consolidated frontends**:

**Before (Broken)**:
```
/etc/haproxy/domains/
├── rentalfixer.app.cfg      # Separate frontend for rentalfixer.app
└── staging.rentalfixer.app.cfg  # Separate frontend for staging
```

**After (Fixed)**:
```
/etc/haproxy/domains/
├── web_http.cfg       # Single HTTP frontend
├── web_https.cfg      # Single HTTPS frontend with ALL certificates
├── web_backends.cfg   # All application backends
└── registry.conf      # Domain → App → Port mapping
```

### Configuration Changes

**web_https.cfg** - Single frontend with multiple certificates:
```haproxy
frontend web_https
    bind :443 ssl \
        crt /etc/haproxy/certs/rentalfixer.app.pem \
        crt /etc/haproxy/certs/staging.rentalfixer.app.pem \
        alpn h2,http/1.1
    mode http
    
    # ACLs for each domain
    acl is_rentalfixer_app hdr(host) -i rentalfixer.app
    acl is_staging_rentalfixer_app hdr(host) -i staging.rentalfixer.app
    
    # Route to backends
    use_backend rentalfixer_backend if is_rentalfixer_app
    use_backend rentalfixer_staging_backend if is_staging_rentalfixer_app
```

### Updated Scripts

**provision-domain.sh** changes:
- Added `--rebuild` flag to rebuild configs without provisioning
- Creates consolidated frontend configs from registry
- Maintains `registry.conf` for domain tracking
- Generates HTTP redirect frontend
- Generates HTTPS frontend with all certificates

### Verification
```bash

curl -sI https://rentalfixer.app          # 200
curl -sI https://staging.rentalfixer.app  # 200

# Both routers work correctly
curl -sk --resolve rentalfixer.app:443:172.93.54.112 https://rentalfixer.app  # 200
curl -sk --resolve rentalfixer.app:443:23.29.118.6 https://rentalfixer.app    # 200
```

**Files Changed**:

### 8. Staging/Production Branch Deployment
**Problem**: Workflow used `develop` branch instead of `staging`.

**Solution**: Updated GitHub workflow generator:
- `main` branch → production deployment
- `staging` branch → staging deployment
- Updated staging app name from `staging-{app}` to `{app}-staging`

**Files Changed**:
- `dashboard/app.py` - `generate_github_workflow()`

### 9. Staging Environment Setup
**Problem**: staging.rentalfixer.app returning 503.

**Solution**: Manually set up staging environment:
- SSL certificate on router-01
- HAProxy config for staging
- App cloned to `/opt/apps/rentalfixer-staging`
- PHP-FPM pool and nginx config
- Database `rentalfixer_staging` created
- APP_ENV=staging set in .env

**Result**: Staging now working at https://staging.rentalfixer.app

### 10. SSL Certificate Provisioning (DNS-01 Challenge)
**Problem**: 
- `provision-domain.sh` used HTTP-01 challenge which doesn't work with Cloudflare proxy
- SSL certs were only provisioned on router-01, not router-02
- Staging CSS not loading due to missing cert on router-02

**Root Cause**: 
- HTTP-01 challenge requires direct HTTP access to the server
- Cloudflare proxy intercepts HTTP traffic, breaking the challenge
- Script only ran on one router instead of both

**Solution**:
1. Updated `/opt/scripts/provision-domain.sh` to use DNS-01 challenge:
   - Uses `certbot certonly --dns-cloudflare`
   - Reads credentials from `/root/.secrets/cloudflare.ini`
   - Falls back to HTTP-01 if credentials not found
   - No need to stop HAProxy during cert generation

2. Script deployed to both routers:
   - `/opt/scripts/provision-domain.sh` on router-01 and router-02
   - Both have Cloudflare DNS credentials

**Files Changed**:
- `/opt/scripts/provision-domain.sh` on both routers

## Key Commands Reference

### HAProxy Management
```bash
# Check HAProxy status
systemctl status haproxy

# Validate configuration
haproxy -c -f /etc/haproxy/haproxy.cfg -f /etc/haproxy/domains

# Reload configuration
systemctl reload haproxy

# Show runtime stats
echo 'show stat' | socat stdio /run/haproxy/admin.sock

# Show loaded certificates
echo 'show ssl cert' | socat stdio /run/haproxy/admin.sock
```

### Domain Provisioning
```bash
# Provision new domain
/opt/scripts/provision-domain.sh domain.tld appname 8100

# Provision with www redirect
/opt/scripts/provision-domain.sh domain.tld appname 8100 --www www.domain.tld

# Provision staging
/opt/scripts/provision-domain.sh staging.domain.tld appname 8101 --staging

# Rebuild all configs from registry
/opt/scripts/provision-domain.sh --rebuild
```

### SSL Certificates
```bash
# List certificates
certbot certificates

# Renew certificates
certbot renew

# Check certificate details
openssl x509 -in /etc/haproxy/certs/domain.tld.pem -noout -text
```
