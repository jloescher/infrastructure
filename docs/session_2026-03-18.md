## Completed Phases

### Phase 23: SSH Timeout Fix

**Issue:** SSH commands had a 30-second timeout cap that caused long-running deploys to fail.

**Fix:** Removed the `min(timeout, 30)` cap in the `ssh_command()` function.

**File:** `dashboard/app.py:430`

### Phase 24: Staging Port Separation

**Issue:** Staging was using production port + 1 (8101), but port 9100 is used by node_exporter.

**Fix:** Implemented separate port ranges:
- Production: 8100-8199
- Staging: 9200-9299

**Files:**
- `dashboard/app.py:88-104` - Port allocation functions
- `dashboard/app.py:1973` - Staging deploy uses `get_staging_port()`

### Phase 25: Domain Health Check Bug Fix

**Issue:** Production deploy was checking ALL domains including staging (which hadn't been deployed yet), causing false failures.

**Fix:** Filter domain health checks by environment:
- Production deploy only checks production domains
- Staging deploy only checks staging domains

**File:** `dashboard/app.py:2462-2478`

### Phase 26: Last Deploy Status Bug Fix

**Issue:** `update_last_deploy_status()` was called before `success_flag` was set, so the flag was always False when saved.

**Fix:** Set `success_flag` before calling `update_last_deploy_status()`.

**File:** `dashboard/app.py:2673-2678`

### Phase 27: Staging Password Protection

**Requirement:** Staging environments must be password-protected via HAProxy basic auth.

**Implementation:**
1. Added `--password` parameter to `provision-domain.sh`
2. Created htpasswd files for staging apps in `/etc/haproxy/htpasswd/`
3. Added HAProxy `http-request auth` with userlist for password-protected backends
4. Updated registry.conf format: `domain=app=port=password`
5. Rebuilt HAProxy config to include auth for staging

**Files:**
- `scripts/provision-domain.sh` - Added password parameter and auth logic
- `dashboard/app.py:3501-3534` - `provision_domain_on_routers()` passes password

**Verification:**
```
# Without credentials: 401 Unauthorized
curl -I https://staging.rentalfixer.app
HTTP/2 401
www-authenticate: Basic realm="Staging Area"

# With credentials: 200 OK
curl -I -u admin:o7YL2YVBEVUl8BaU https://staging.rentalfixer.app
HTTP/2 200
```

### Phase 28: UI Bug Fix - Domain Provisioning Reason

**Issue:** Domain provisioning phase always showed "(deploy failed)" when skipped.

**Fix:** Display actual reason. For staging deploys, show "N/A" badge with clear explanation.

**File:** `dashboard/templates/app_status.html:60`

### Phase 29: GitHub Webhook Timeout Fix

**Issue:** GitHub has a 10-second webhook timeout, but deployments take several minutes.

**Fix:** Run deploys asynchronously:
- Return `202 Accepted` immediately
- Run deploy in background thread using Python `threading`
- Deploy status stored in `applications.yml`

**Response:**
```json
{
  "success": true,
  "message": "Deploy started",
  "branch": "staging",
  "environment": "staging"
}
```

### Phase 30: Last Deploy Status on Applications List

**Issue:** Applications list didn't show last deploy status, environment, or branch.

**Fix:**
- Updated `update_last_deploy_status()` to store branch and environment
- Updated `apps.html` to show success/failed badge with timestamp, environment, and branch
- Replaced "Database" column with "Last Deploy" column

**Format:** `[success/failed] 2026-03-18 03:34` followed by `production (main)`

## Additional Fixes

### Router-02 Webhook Routing Missing

**Issue:** Router-02 was missing hooks routing ACLs, causing 404 errors when GitHub hit that router.

**Fix:** Rebuilt HAProxy config on router-02.

### PHP-FPM Metrics Not Working

**Issue:** `pm.status_path` was disabled in www.conf, causing empty PHP-FPM metrics.

**Fix:** Enabled `pm.status_path = /status` in `/etc/php/8.5/fpm/pool.d/www.conf` on both app servers.

### Webhook Secret Not Visible

**Issue:** Webhook secret was only shown on app creation result page.

**Fix:** Added webhook secret display to App Status page under Webhook URL.
